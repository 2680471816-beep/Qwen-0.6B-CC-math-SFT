# 数据处理 Pipeline 设计方案

**项目**：Qwen3-0.6B-Base 数学推理后训练
**目标**：从 100K 条原始网页数学文本中构建高质量 SFT 训练集
**预期输出**：40,000~45,000 条标准 QA pair

---

## 总体流程

```
原始数据 (100K JSONL)
    │
    ▼
[Stage 1] 快速过滤（元数据 + 规则）      → ~40K 条
    │
    ▼
[Stage 2] 文本清洗（去除结构性噪声）      → ~38K 条
    │
    ▼
[Stage 3] 内容类型分类                   → 分4路处理
    │
    ├── 单题解答型 (~24%)  → [4A] 规则提取 + 少量 LLM
    ├── 论坛 QA 型 (~6%)   → [4B] 半结构化 LLM 提取
    ├── 文章教程型 (~62%)  → [4C] 隐含例题挖掘 + LLM 生成
    └── 教科书型 (~8%)     → [4D] 切分 + LLM 生成
    │
    ▼
[Stage 4] QA 生成与提取                  → ~45K 候选对
    │
    ▼
[Stage 5] 质量过滤与验证（三层机制）     → ~32K 条
    │
    ▼
[Stage 6] 去重 + 多样性采样 + 格式化     → 40~45K 条最终训练集
```

---

## Stage 1：快速过滤

**方法**：纯规则，无需 GPU
**输入**：100,000 条 → **输出**：约 40,000 条

### 过滤规则

| 规则 | 阈值 | 说明 |
|------|------|------|
| 文本长度下限 | < 800 字符丢弃 | 内容太短，无实质数学内容 |
| 文本长度上限 | > 50,000 字符单独存入「超长队列」 | 教科书级长文，后续切分处理 |
| nemocurator_scores | < 1.5 丢弃 | 教育质量过低，主要排除 0分/低分 文档 |
| 噪声密度 | 每100字符噪声匹配 > 0.15 丢弃 | 导航菜单、广告、论坛元数据等 |
| 数学内容密度 | 每500字符数学指标 < 0.5 丢弃 | LaTeX 公式、数学动词等指标稀疏 |

> **为什么 nemocurator 阈值设为 1.5 而非 2.0？**
> 54.7% 的数据得分为 2，直接卡在整数边界会丢弃大量中等质量数据。1.5 作为软边界，既排除明显噪声（0~1分共 15.4%），又保留有价值的 1 分文档中的优质部分。

**噪声检测关键词（正则）：**

```
导航菜单：(Overview|Reviews|Salaries|Benefits|Interviews){3,}
CMS 标记：\[INS:.*?:INS\] 或 \[DEL:.*?:DEL\]
论坛元数据：Joined:\s*\d | Posts:\s*\d | Thanks:\s*\d
视频时间码：\d+m\s+\d+s
广告声明：Cookie | Subscribe | Advertisement | Sponsored
```

---

## Stage 2：文本清洗

**方法**：规则 + 正则表达式，无需 GPU
**输入**：约 40,000 条 → **输出**：约 38,000 条

### 清洗操作

| 操作 | 方法 | 备注 |
|------|------|------|
| 删除导航/界面元素 | 正则匹配 Home/About/面包屑/页码/分享按钮 | 保留数学内容行 |
| 删除 CMS 编辑标记 | `[INS:...:INS]` 保留内容去掉标记；`[DEL:...:DEL]` 整体删除 | 常见于内容管理系统 |
| 删除论坛元数据块 | 匹配 Joined/Posts/Thanks 行 | 不删回帖内容，只删用户信息 |
| **删除错误答案段落** | 匹配 WRONG/incorrect answer/错误解法 等标注的段落 | **见下方重点说明** |
| 标准化数学公式格式 | `\(...\)` → `$...$`；`\[...\]` → `$$...$$` | 统一 LaTeX 分隔符 |
| 压缩多余空行 | 连续3行以上空行压缩为2行 | 减少格式噪声 |
| 清洗后长度校验 | 有效内容不足 600 字符则丢弃 | 防止清洗后变空壳 |

> **为什么要主动删除错误答案段落？**
> 部分文档包含被明确标注为"错误"的解题过程（如竞赛题讨论中的错误尝试）。
> 对于 0.6B 小模型，记忆容量有限，**SFT 时无法区分"这是反例"和"这是正确做法"**，错误推理路径会直接覆盖正确模式。因此在数据层面主动删除，而非依赖模型自行甄别。
> 注意：仅删除被明确标注为错误的段落，"先展示错误再纠正"的完整教学段落在 Stage 5 的质量验证中处理。

---

## Stage 3：内容类型分类

**方法**：启发式规则打分，无需 GPU
**输入**：约 38,000 条 → **输出**：按类型分流至 4 个队列

> **为什么不用大模型做分类？**
> 用小模型（Qwen3-0.6B）分类效果差；用大模型成本极高（38K 条 × 分类 prompt ≈ 大量 token）。
> 规则分类虽然召回不完美，但精度足够，且完全可审计、可调整。

### 分类结果与处理路由

| 内容类型 | 特征识别 | 预期数量 | 后续处理 |
|----------|----------|----------|----------|
| **单题完整解答型** | 含 Problem/Solution/Answer 等结构标志 | ~9,100 条 | Stage 4A |
| **论坛 QA 型** | 含 Posted by/Reply #/Quote: 等论坛标志 | ~2,280 条 | Stage 4B |
| **文章教程型** | 含章节标题/定义/定理，无明确 QA 结构 | ~23,560 条 | Stage 4C |
| **教科书型** | 超长文本 + 章节密集结构 | ~3,140 条 | Stage 4D |
| **计算器工具页** | 大量重复数字变体，数字重复率极高 | ~760 条 | **直接丢弃** |
| **其他** | 视频讲义/教案等，无法归类 | ~1,140 条 | **直接丢弃** |

---

## Stage 4：QA 生成与提取（核心阶段）

这是唯一大量消耗 GPU 资源的阶段，使用 **Qwen3-0.6B-Base** 本身完成 QA 生成。

### 4A：单题解答型 — 直接提取

**输入**：~9,100 条 → **输出**：~8,000 条

> **为什么优先规则提取而非重新生成？**
> 这类文档本身已有完整的 problem + solution，直接提取比重新生成更可靠，**不引入幻觉**，且这是质量最高的数据，应最大化保留。

**两步处理：**

1. **规则提取**（约60%可直接提取）：识别 `Problem...Solution...` 等明确分隔结构
2. **LLM 辅助提取**（约40%结构不清晰的）：

```
从以下文本中提取数学问题和完整解答。

要求：
- 问题：只提取数学题干，1-3句话
- 解答：包含推导步骤和明确最终答案的完整解题过程
- 若有多个问题，只提取第一个完整的
- 若无法找到明确的问题-解答对，输出：SKIP

文本：{text}

输出格式：
QUESTION: [题目]
SOLUTION: [分步解答]
```

---

### 4B：论坛 QA 型 — 半结构化提取

**输入**：~2,280 条 → **输出**：~1,600 条（提取率约 70%，论坛噪声较多）

**处理策略**：识别主帖中的数学问题 + 回帖中最完整的解答

```
从以下论坛讨论中提取干净的数学问答。

要求：
- 问题：从原帖提取数学题干（忽略个人寒暄）
- 解答：选取回帖中最清晰、有步骤的解答
- 若有多个解答，优先选有逐步推导的
- 忽略仅含"我同意"/"好问题"等无数学内容的回复
- 若题目不清晰或无有效解答，输出：SKIP

论坛帖子：{text}

输出：
QUESTION: [数学题目]
SOLUTION: [最佳解答，分步骤]
```

---

### 4C：文章教程型 — 挖掘式生成

**输入**：~23,560 条 → **输出**：~18,000 条（最大数量来源）

**两阶段处理：**

**阶段一：规则挖掘隐含例题**（约40%的文章含此类内容）

识别 `For example / Example N / Consider the following / Let us find` 等引导词后跟解答的段落，规则可提取约 **9,400 对**。

**阶段二：LLM 生成配套练习题**（对剩余约 14,160 篇）

> **为什么要生成练习题而不直接使用文章原文？**
> 文章教程型文本直接作为 SFT 数据有三个问题：
> ① 格式不是 QA，无法直接训练推理能力
> ② 含大量定义、过渡语句，小模型难以从中学习推理模式
> ③ 原文长度往往超过模型最优输入范围
> 因此转化为针对性练习题，将知识点变为可训练的推理过程。

```
请根据以下数学教程文章，出一道配套练习题，并给出完整解答。

出题要求：
- 必须是具体的数值或代数计算题，不是概念性问题
- 难度：中等（使用文章介绍的方法即可解决）
- 题目长度：2-4句话

解答要求：
- 恰好 3-5 个清晰的推理步骤
- 每步推进求解
- 结尾明确给出最终答案
- 解答总长：150-300词

文章内容（节选）：
{text_chunk}  ← 只传核心段落，最多2000字符

输出：
PROBLEM: [出的题目]
SOLUTION: [分步解答，含最终答案]
```

**文章切块策略**：
- 按章节/段落切分为 1,500~2,000 字符的块
- 每块生成 1 个 QA pair
- 同一文章**最多生成 2 个** QA pair，防止过拟合

---

### 4D：教科书型 — 切分 + 生成

**输入**：~3,140 条（含超长队列） → **输出**：~5,000 条

按标题（`##`/`###`）优先切分，块大小 1,500 字符，重叠 200 字符（防止截断公式）。

```
从以下数学教材片段中，提取或创建一道练习题。

优先策略：
1. 若片段中已有题目，直接提取
2. 若无题目，针对核心定理或方法创建一道题

要求题目 + 解答必须自洽（不得引用"如上所示"等指代）。

教材片段：
{chunk}

输出：
PROBLEM: [自洽的题目描述]
SOLUTION: [分步解答，3-5步，含最终答案]
```

---

### Stage 4 汇总

| 来源 | 输入 | 输出 | 主要方法 |
|------|------|------|----------|
| 4A 规则提取 | ~5,460 | ~5,000 | 正则规则 |
| 4A LLM 辅助提取 | ~3,640 | ~3,000 | LLM |
| 4B 论坛提取 | ~2,280 | ~1,600 | LLM |
| 4C 规则挖掘 | ~23,560 | ~9,400 | 正则规则 |
| 4C LLM 生成 | ~14,160 | ~8,500 | LLM |
| 4D 教科书生成 | ~3,140 | ~5,000 | LLM |
| **合计** | | **~32,500** | |

## Stage 5：质量过滤与验证

**输入**：约 32,500~45,000 条候选 QA → **输出**：约 32,000 条

这是整个 pipeline 中**最关键的质量控制环节**，三层防御机制确保错误答案不进入训练集。

### 5.1 格式完整性检查（规则）

| 检查项 | 规则 |
|--------|------|
| 问题长度 | 30~1,000 字符 |
| 解答长度 | 100~3,000 字符 |
| 包含数学内容 | 含 `$`、`\`、`=`、数字等数学符号 |
| 有明确最终答案 | 解答末尾含 `= 数字`、`\boxed`、`answer is`、`therefore` 等 |
| 未被截断 | 解答不以 `...`、`and`、`,` 结尾 |
| 问题是问题 | 含 `?` 或 find/prove/solve/compute 等动词 |

预期通过率：约 **85%**

---

### 5.2 数学答案三层验证（核心机制）

> **为什么需要三层验证？**
> 错误答案对 0.6B 小模型的危害远大于大模型。大模型有足够参数区分正误，而小模型会直接记忆错误的推理路径。宁可少数据，也要保证正确率。

**第一层：符号计算验证（sympy）**

针对含数值结果的题目，用 sympy 从题目重新求解并与答案比对：
- 验证结果为 `correct` → 保留
- 验证结果为 `wrong` → **直接丢弃**
- 无法验证（sympy 无法处理）→ 进入第二层
- 覆盖范围：约 30% 的数据（简单代数/算术题）

**第二层：LLM 交叉验证**

针对 sympy 无法验证的高优先级样本（4B/4C/4D 生成内容），使用**验证器模型**（与生成器不同，避免一致性偏差）：

```
你是一个数学答案检查员，请严格验证以下解答是否正确。

检查要点：
1. 每一步推导是否逻辑合理？
2. 算术/代数运算是否正确？
3. 最终答案与过程是否一致？

问题：{question}

待验证解答：
{solution}

请回答：
VERDICT: CORRECT / INCORRECT / UNCERTAIN
REASON: [1-2句说明原因]
如果 INCORRECT，请简述错误所在。
```

策略：
- 4A（直接提取）**跳过此步**，置信度已够高
- 4B/4C/4D 生成内容：**随机抽取 30%** 进行验证
- `INCORRECT` 的样本：若问题有价值，可尝试重新生成解答；否则丢弃

**第三层：来源可信度统计惩罚**

若同一来源文章生成的多个 QA 中有一个被判 `INCORRECT`，则提升该文章所有 QA 的审查权重（来源可信度惩罚）。

---

### 5.3 长度与复杂度适配检查

> **针对 0.6B 小模型的 token 预算约束**
> SFT 最优总长（question + solution）：**300~800 tokens**（约 1,200~3,200 字符）
> 简短 CoT（3~5步）是性价比最优格式：小模型生成超长 CoT 容易出错，且会占用推理 token 预算。

| 情况 | 处理方式 |
|------|----------|
| 估算 token 数 < 150 | 标记为过短，丢弃 |
| 估算 token 数 300~800 | 最优，直接保留 |
| 估算 token 数 > 900 | 截断策略：保留前3步 + 最终答案步骤 |

---

### 5.4 近似去重（TF-IDF）

对所有 question 做 TF-IDF 向量化，余弦相似度 > 0.85 的视为近似重复，保留质量分更高的一条。

### Stage 5 汇总

| 过滤项 | 预期通过率 | 丢弃量 |
|--------|------------|--------|
| 格式完整性 | 85% | ~6,700 |
| sympy 验证（判错） | 97% 通过 | ~1,200 |
| LLM 验证（判错） | 88% 通过 | ~3,500 |
| 近似去重 | 95% 通过 | ~1,600 |
| **综合输出** | | **约 32,000 条** |

---

## Stage 6：去重 + 多样性采样 + 格式化

**输入**：约 32,000 条 → **输出**：40,000~45,000 条最终训练集

### 6.1 精确去重（MD5 哈希）

对 question 字段做小写标准化后计算 MD5，去除完全重复条目。

### 6.2 数学主题多样性采样

> **为什么需要强制主题分布？**
> 原始数据中代数类题目占比极高，若不干预，模型会在代数上过拟合，而在几何、数论等子域能力退化，影响测试集成绩。

参考 MATH benchmark 的子域分布：

| 数学主题 | 目标占比 | 识别关键词 |
|----------|----------|------------|
| 代数 | 30% | equation / polynomial / solve for / linear / quadratic |
| 几何 | 15% | triangle / circle / area / volume / angle |
| 微积分 | 15% | derivative / integral / limit / differentiate |
| 数论 | 15% | prime / divisor / modulo / gcd / remainder |
| 统计概率 | 10% | probability / mean / variance / distribution |
| 组合计数 | 10% | permutation / combination / counting |
| 其他 | 5% | 不属于以上类别 |

### 6.3 最终 SFT 格式化

**主流格式（95% 数据）—— 简短 CoT**

> **为什么不用 `<think>` 标签作为主流？**
> 0.6B 模型对格式非常敏感，复杂格式（`<think>` + 答案分离）增加了学习难度，让模型分散注意力到格式学习上。简洁 CoT 让模型专注于数学推理本身。

```json
{
  "messages": [
    {
      "role": "user",
      "content": "Find the value of x if 3x + 7 = 22."
    },
    {
      "role": "assistant",
      "content": "Subtract 7 from both sides: 3x = 15.\nDivide both sides by 3: x = 5.\nThe answer is x = 5."
    }
  ]
}
```

**实验格式（5% 数据）—— thinking 格式（用于 ablation）**

```json
{
  "messages": [
    {
      "role": "user",
      "content": "Find the value of x if 3x + 7 = 22."
    },
    {
      "role": "assistant",
      "content": "<think>\nI need to isolate x.\n3x + 7 = 22\n3x = 22 - 7 = 15\nx = 15/3 = 5\n</think>\nx = 5"
    }
  ]
}
```

| 阶段 | 主要方法 | 预计 GPU 时（A100） |
|------|----------|---------------------|
| Stage 1：快速过滤 | CPU 规则 | 0 |
| Stage 2：文本清洗 | CPU 规则 | 0 |
| Stage 3：内容分类 | CPU 规则 | 0 |
| Stage 4：QA 生成 | LLM 推理（本地模型） | **约 6 小时** |

## 关键设计决策总览

| 决策 | 核心理由 |
|------|----------|
| nemocurator 阈值设为 1.5（而非整数 2） | 避免在整数边界丢弃大量中等质量数据，软边界更合理 |
| 内容分类用规则（而非大模型） | 成本低、可审计，精度已满足分流需求 |
| 单题解答型优先规则提取（而非重新生成） | 人类写的 solution 比 LLM 生成更可靠，减少幻觉 |
| 文章型用 LLM 生成 QA（而非直接用原文） | 原文格式不适合小模型 SFT，转化为标准 QA 效果更好 |
| 三层答案验证（sympy + LLM + 统计） | 错误答案对 0.6B 小模型危害极大，宁少勿错 |
| solution 控制在 3~5 步 | 小模型生成超长 CoT 易出错，短 CoT 是性价比最优格式 |
